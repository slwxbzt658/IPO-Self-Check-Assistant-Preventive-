HawkEye 的架构设计解决一个核心矛盾：AI 需要足够的推理自由才能发现未知风险，但又必须被严格约束才能保证输出可信。我们用分层 Harness 来处理这个矛盾。

------

### 如何读这张图

这张图使用**嵌套结构**而非箭头，原因是：嵌套关系本身就表达了约束——外层必须在内层划定的空间内活动，越靠内层约束力越强、越不可改变，越靠外层灵活性越高。

五个概念从内到外的顺序及逻辑：

```
硬 Harness（最内层）
  → 标准化审查协议（在法律框架内固化的检查单）
    → 推理范式库（在协议框架内积累的案例知识）
      → CoT-RAG（在范式库基础上运作的检索机制）
        → 结构化推理提示词工程（最外层，组装一切的实现手段）
```

**为什么范式库在 CoT-RAG 内层？**

推理范式库是知识本身（装着什么案例），CoT-RAG 是检索机制（怎么找到案例）。知识比检索机制更基础，所以范式库在内层，CoT-RAG 在其外层运作。

## 框架分层

| 层次                       | 内容                                | 性质                         |
| :------------------------- | :---------------------------------- | :--------------------------- |
| Harness 第一层（法律框架） | 6 大支柱、BGOR 色码、三区分层       | 普适的，任何 A 股公司都适用  |
| Harness 第二层（探针规则） | 客户集中度 > 70% 触发、6 维订单审查 | 半普适的，来自裕鸢案例的归纳 |

## 核心概念

#### 硬 Harness（不可逾越的边界）：

  - 6 大支柱的法律定义
  - 输出必须带证据溯源
  - 不能发表投资建议
    → 无论什么公司、什么时代，这些都成立
  - 是系统里最外层的护栏

#### 软 Harness（基于案例学习的推理约束）：

  - 客户集中度探针（从裕鸢归纳）
  - 6 维订单审查（从裕鸢归纳）
    → 随着矩阵公司扩充，这部分会不断演化
  - 会随着agent的进化而演化，属于中间层
  - 软 Harness = 模型穿透里每一个小项的判断标准，这些标准来自案例积累、行业惯例、法规要求，写进 JSON 文件里，可以随着新案例不断更新

#### Chain-of-Thought RAG（思维链增强的检索生成）

<img src="D:\AI_Projects\hawkeye-ipo-v5\docs\whitepaper\images\COT-RAG_v0.1.png" style="zoom:67%;" />

**把探针的生成本身交给 LLM，但 LLM 的推理空间被硬 Harness 的法律框架约束。这是检索范式 + LLM 模仿推理，在系统中是运行时的推理机制。**

- 硬边界：你只能在这 6 个支柱的法律框架内思考
- 软探针：根据这家公司的具体信号，LLM 自主决定"需要追查哪些维度"
- 推理链：从知识库里找最相似的案例，模仿它的追查方式

#### 标准化审查协议（Standardized Audit Protocol）:触发即固定的检查单

​	系统分三层，每层对应不同的稳定性：

```
		第一层：审查协议（最稳定，不因公司而变）
​			→ 一旦识别到"客户依赖"问题，触发固定的12项检查
 			→ 一旦识别到"科创属性"问题，触发固定的8项检查
  			→ 检查项对所有公司一致，像体检套餐

​		第二层：事实输入（因公司而变）
  			→ 裕鸢的检查单里：客户集中度70.43%、销售4人
  			→ XX公司的检查单里：客户集中度35%、销售12人
  			→ 同样的格子，填不同的数字

​		第三层：结论判定（因事实而变）
​			→ 裕鸢：12项里8项异常 → 红色
 			→ XX公司：12项里2项异常 → 黄色
```

​				在系统中是软 Harness 的具体形态

#### 结构化推理的提示词工程（Prompt Engineering for Structured Reasoning）

让以上一切协同工作的技术，是系统中的实现层

Chain-of-Thought Scaffolding（思维链脚手架）：概念的这个图。每一步提示词，都是在给 LLM 搭一层脚手架，让它的推理在你控制的轨道上一步步走下去。

Few-shot 与推理范式库的关系：每次分析新公司时，系统从推理范式库中检索最相似的历史案例，将其推理链作为 Few-shot 示例注入提示词，让 LLM 模仿已验证的推理方式，而不是从零开始生成。



硬 Harness     = 医院的医疗伦理法规（永远不能违反）
软 Harness     = 各科室的诊疗规范（基于积累，会更新）
标准化审查协议 = 具体某种病的标准检查套餐
推理范式库     = 医院的病历库（积累的典型案例）
CoT-RAG        = 医生查阅相似病历后给出诊断的过程
提示词工程     = 培训医生"如何使用病历库做诊断"的方法

## 两个视角读同一套架构

同一张嵌套结构图，可以从两个不同的维度来理解：

### 系统执行视角（一个新案子进来时发生了什么）

描述的是系统**运行时**的工作流程，回答"处理一家新公司时，各层如何协同"：

```
新公司招股书进入
      ↓
[CoT-RAG] 从推理范式库检索最相似的历史案例
      ↓
[标准化审查协议] 识别风险类型，触发对应固定检查单
      ↓
[结构化推理提示词工程] 将检索案例 + 检查单 + 公司数据组装为 Prompt
      ↓
LLM 生成推理报告
      ↓
[硬 Harness] 过滤输出：无溯源结论删除，投资建议删除
      ↓
输出四色笔记
```

关键特征：硬 Harness 在执行末端做最终过滤，但它的规则从第一步就写入了 System Prompt，贯穿全程。

---

### 系统进化视角（随着案例积累，系统怎么变聪明）

描述的是系统**随时间演化**的路径，回答"这套知识体系是怎么从无到有建起来的"：

```
第一阶段：人工分析第一个案例（裕鸢，NotebookLM 协作）
      ↓
第二阶段：将推理链条整理为统一格式，存入推理范式库
      ↓
第三阶段：持续分析矩阵公司，范式库案例数量增长
      ↓
第四阶段：从库中发现跨案例规律
          "凡是客户集中度问题，必须查这 12 项"
          → 此类规律固化为标准化审查协议
      ↓
第五阶段：范式库足够厚之后，CoT-RAG 检索准确率显著提升
          新公司进来，系统自动找到最相关的参考案例
      ↓
系统达到：见过的风险类型越多，新案子的分析质量越高
```

关键特征：硬 Harness 不是进化链的终点，而是**贯穿所有阶段的不变底线**。第一天建立，最后一天仍然有效。

---

**两个视角的关系：**

> 进化视角解释系统如何被建造；执行视角解释系统如何运行。
> 两者使用同一套架构，只是观察的时间维度不同。

## 系统质量保障：人在回路的反馈循环

### 为什么规则和检查单不够

**标准化审查协议（方式二）的局限：**
检查单只能覆盖**已知的风险类型**。第一次出现的新型商业风险，检查单里没有这一项，系统直接漏掉。

**Hard Harness 强制规则（方式一）的局限：**
无法预先列举所有需要建模的情况。商业逻辑是无穷的，写死的规则永远有盲区，无法做到普适化。

### 核心解决机制：人工审核 + 范式库补充

```
┌─────────────────────────────────────────────────────┐
│                  人在回路反馈循环                      │
│                                                      │
│  Agent 分析新公司                                     │
│       ↓                                              │
│  输出四色笔记（可能有盲区）                             │
│       ↓                                              │
│  人工专家审核（NotebookLM 协作 + 人工判断）             │
│       ↓                                              │
│  发现 Agent 遗漏的关键穿透点？                          │
│    ├── 是 → 人工补充深度分析                           │
│    │        → 将此次推理链条整理为统一格式              │
│    │        → 存入推理范式库                           │
│    │        → 下次类似案例，CoT-RAG 自动检索到          │
│    │        → Agent 自动完成同等深度分析                │
│    └── 否 → 本次分析质量合格，直接归档                  │
└─────────────────────────────────────────────────────┘
```

### NotebookLM 与 Agent 的分工

这两者不是竞争关系，而是喂养关系：

| | NotebookLM + 精炼提示词 | HawkEye Agent |
|---|---|---|
| **优势** | 灵活，能识别新型商业风险，能做全模型穿透 | 速度快，一致性强，可规模化 |
| **局限** | 每次需要人工在场引导，无法自动化 | 遇到新类型问题时分析深度不足 |
| **角色** | 知识生产者 | 知识执行者 |

**核心原则：**
> NotebookLM 协作产出的深度分析 → 人工整理后进入范式库 → Agent 下次自动复现同等深度

你与 NotebookLM 打磨提示词越深、积累的案例越多，Agent 将来能自动处理的情况就越广、分析越深。

### 为什么这是可持续的

系统不依赖"一开始就写出完美规则"，而是依赖**持续的人机协作积累**：

```
第 1 个案例（裕鸢）：大量人工干预，产出深度分析，存入范式库
第 5 个案例：部分类型 Agent 已能自动处理，人工只需审核新类型
第 20 个案例：Agent 覆盖大多数已知风险类型，人工主要处理边缘案例
第 N 个案例：Agent 趋近专家级分析，人工仅处理真正的前所未见情况
```

这是系统设计的核心哲学：**不追求一次性完美，追求可持续进化。**

------

## 系统定位：不是替代专家，而是解放专家

HawkEye 不是要模拟一个全知全能的发审委，而是把**已知的、可归纳的审查逻辑系统化**，让人工专家只需要面对真正新颖的、无先例的问题。

```
没有 HawkEye 的分析师：
  → 靠经验和记忆覆盖所有检查点
  → 容易在已知风险上遗漏，精力也消耗在重复性工作上

有 HawkEye 的分析师：
  → 已知风险类型由系统强制触发、量化建模、交叉验证
  → 分析师的精力集中在系统识别不了的新型风险上
```

正确的评估基准不是"AI vs. 完美发审委"，而是"有 AI 辅助的分析师 vs. 没有 AI 辅助的分析师"。

## 系统适应性：如何应对现实世界的变化

现实世界持续变化：法规更新、商业环境演变、新型造假手段出现、发审委审核思路迭代。HawkEye 的架构通过以下机制保持适应性：

| 变化类型 | 对应更新机制 | 更新触发方 |
|---|---|---|
| 法规条文修订 | 更新 Hard Harness 规则（rules_schema.json 版本升级） | 人工审核后批准 |
| 新型造假手段 | 人工分析新案例后加入推理范式库 | 人工创建，系统自动生效 |
| 发审委新审核思路 | 复盘新失败案例，更新标准化审查协议 | 人工更新协议条目 |
| 行业阈值基准变化 | 修改 Soft Harness 的 JSON 配置 | 不需要改代码，直接修改配置文件 |

**重要设计选择：系统不自主更新判断标准。**

金融审核具有法律责任，判断标准的任何变更必须经过人工确认后才能生效。这不是技术局限，而是合规要求下的主动设计决策。

## Human-in-the-Loop：永久性架构特征，而非临时补丁

人工介入回路（Human-in-the-Loop）是 HawkEye 的核心架构特征，不会随系统成熟而消失，原因如下：

**1. 法律责任归属**
IPO 审核结论涉及法律责任，不能完全委托给 AI。系统输出的是"辅助判断"，最终结论由人工专家背书。

**2. 新型风险的首次识别**
范式库里没有先例的风险类型，只有人能第一次识别。人工专家是系统"见识"的来源。

**3. 知识库质量把关**
LLM 生成的范式库草稿须经人工审核批准后才能入库，防止低质量样本污染知识库。

**4. 幻觉与错误的最终过滤**
即使 Hard Harness 已过滤大部分问题，人工复核仍是最后一道防线。

```
人工介入的四个节点：
  ① 深度分析阶段：NotebookLM 协作，发现系统盲区
  ② 范式库入库前：审核 LLM 生成的推理链草稿
  ③ 规则更新时：批准 Hard/Soft Harness 的任何变更
  ④ 高风险结论输出后：最终判断由人工专家签字确认
```

## 推理范式库的内容边界：三区分层与多轮问询的关系

范式库不是把所有分析内容都装进去，它的内容边界由三个区域严格划定。

### 三个区域回顾

```
区域 A（招股书）   → Agent 自动运行的主战场，只用公开静态数据
区域 B（多轮问询） → 验证区域 A 预测准确率的质检层
区域C（底层核查底稿）→ 高级专家模式，V6+ 路线图
```

### 多轮问询内容如何分流进入范式库

多轮问询函里存在两类性质完全不同的内容，必须分别处理：

**第一类：发审委追问的方向本身**

```
发审委追问："请说明专家顾问费的支付对象及金额"
      ↓
反推逻辑：招股书里一定有某个信号触发了这个追问
          → 制造费用偏高 + 销售人员极少 = 触发信号
      ↓
进入区域 A 范式库：
  "制造费用偏高 & 销售人员极少 → 必须追查顾问费去向"
      ↓
Agent 下次看到类似信号，主动追这个方向 ✓
```

这类内容**应该进入区域 A 范式库**——它教 Agent 看到什么信号就往哪个方向深挖。

**第二类：被追问后才暴露的证据**

```
企业回复披露：石勇实为新增客户 D01 单位员工
      ↓
这个事实在原版招股书里找不到
      ↓
不进区域 A 范式库（Agent 在招股书阶段根本拿不到这个数据）
      ↓
进入区域 B 验证层：
  "Agent 预警了获客异常（D2维度 Fail）"
  "多轮问询证实了裙带关系"
  → 记录为：预警有效，准确率 +1 ✓
```

这类内容**只进区域 B 验证层**——它不能帮 Agent 在招股书阶段发现问题，但能证明 Agent 的预警是准确的。

### 分流原则

> **"发审委追问的方向"** → 进区域 A 范式库
> （教 Agent：看到什么信号，就往哪里追）
>
> **"被追问后才暴露的证据"** → 进区域 B 验证层
> （证明 Agent 的预警有效，用于 Eval 指标统计）

这个区分决定了范式库的信息质量。混入区域 B 内容会导致 Agent 在招股书阶段试图寻找根本不存在的证据，产生幻觉。

### 三区框架的双重用途

三区框架最初的设计目的是**数据架构**——划定系统在不同阶段能使用什么数据、哪些功能面向什么用户：

```
区域 A → 只用招股书（MVP 核心，面向所有用户）
区域 B → 加入多轮问询（验证层，面向内部 Eval）
区域 C → 底层核查底稿（未来专家模式，V6+）
```

但同一个框架同时也自然划定了**范式库的内容边界**：

```
区域 A 的推理内容   → 可以进范式库（招股书里能找到证据）
区域 B 的方向信号   → 可以进范式库（反推出招股书里的触发信号）
区域 B 的隐藏证据   → 不能进范式库（招股书里根本不存在）
区域 C 的全部内容   → 不能进范式库（完全超出招股书范围）
```

这是同一个框架的两种读法：

| 视角 | 三区的作用 |
|---|---|
| 数据架构视角 | 划定系统在不同阶段能使用什么数据 |
| 知识管理视角 | 划定什么内容能进入范式库、什么不能 |

三区框架的稳固性，正来源于它同时满足了这两个用途，而不是为其中一个单独设计的。

## Agent 演化路线图

**当前阶段（V5）：单案例 MVP**
- 覆盖支柱一（独立性）的主要风险类型
- 范式库基于裕鸢单一案例
- 人工介入程度高

**近期目标（V5.x）：矩阵扩充**
- 分析覆盖六大死因的矩阵公司（目标：每种死因至少 2-3 个案例）
- 范式库案例数量显著增长
- Agent 自动处理比例提升，人工介入集中在新型风险

**中期目标（V6）：跨支柱完整覆盖**
- 支柱二（创新性）和支柱三（成长性）从 Stub 升级为真实模型
- 标准化审查协议覆盖所有六大支柱的主要风险类型
- 具备初步的自动法规解析能力（新法规文本 → LLM 提取关键条款 → 人工批准后更新 Hard Harness）

**长期目标（V6+）：自适应预审系统**
- 系统主动标记低置信度案例，请求人工介入
- 每次人工介入自动生成范式库候选条目
- 覆盖创业板、科创板等多板块差异化审核规则
