# HawkEye · IPO 审核风险预警系统 v5

> 通过反向解构 A 股被否案例，还原发审委审核逻辑，构建可自动运行的 IPO 预审风险分析引擎。

---

## 项目背景

A 股 IPO 审核有一套严格的法律框架。任何企业被否，原因都不超出 **6 大核心支柱**（独立性、科创属性、持续经营等）的范畴。

本项目以**成都裕鸢航空智能制造股份有限公司**（2023 年创业板被否）为解剖样本：

- 逐页拆解招股书 + 三轮问询函
- 还原发审委从"表象数据"到"终极定性"的完整审核逻辑
- 将审核逻辑工程化为可自动运行的 AI 探针系统

---

## 系统架构

```
招股书 PDF
    ↓
多策略抽取器（4 种策略 + 合理性过滤）
    ↓
支柱一·独立性审查（编排者）
    ├── L1：客户集中度穿透探针
    ├── L2：订单获取正当性 6 维审查
    └── 跨支柱 API 调用
          ├── → 支柱二·技术不可替代性（Stub）
          └── → 支柱三·议价权与稳定性（Stub）
    ↓
综合判决 + 跨区域搜查令预埋
    ↓
结构化 JSON 输出（含原文证据 + 法规依据）
```

### 三区数据分层

| 区域 | 数据来源 | 用途 |
|---|---|---|
| **区域 A**（已实现） | 招股书 PDF | 预审 MVP，上线即用 |
| **区域 B** | 多轮问询函 | Eval 验证，测试 A 区预测准确率 |
| **区域 C**（路线图） | 非公开底稿 | 券商内核部深度尽调 |

## 核心设计亮点

**1. 规则与代码解耦**
所有判定阈值存在 JSON 配置文件中，业务规则变更不需要改代码。

**2. 编排者模式（Orchestrator Pattern）**
支柱一不直接做技术或财务评估，而是通过 API 调用支柱二、支柱三，实现微服务式跨支柱协作。

**3. 证据强制溯源**
每一条风险结论都必须附带招股书原文片段 + 页码 + 法规依据，杜绝 AI 幻觉。

**4. 多策略 PDF 抽取**
4 种定位策略（重大事项提示 / 第一大客户锚点 / 客户名精确匹配 / 章节兜底）+ 合理性过滤（自动淘汰"前五合计"等误匹配），实测命中率高。

## 方法论：BGOR 四色推演

每个微创模型采用统一的四色推演框架：

- 🔵 **Blue**：法规裁判标准（法律依据）
- 🟢 **Green**：招股书事实抓取（原文证据）
- 🟠 **Orange**：模型推演 + API 调用
- 🔴 **Red**：层级结论 + 终极定性
